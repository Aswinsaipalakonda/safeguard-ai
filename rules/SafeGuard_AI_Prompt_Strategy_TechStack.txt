⚡  THE VERDICT: Break-Into-Phases Wins Every Time
Giving an AI coding agent one massive prompt causes context overload, code drift, hallucinated APIs, and half-baked features. Breaking into 6 focused phases gives the agent a clear mental model per phase — producing cleaner code, fewer errors, and a working app faster.
1. Single Prompt vs. Break-Phases — The Real Answer
❌  SINGLE GIANT PROMPT
✅  BREAK-INTO-PHASES (Recommended)
✗  Agent loses track of context midway
✓  Each prompt = one clear responsibility
✗  Hallucinated imports, wrong API calls
✓  Agent can verify each phase before moving on
✗  Frontend built without knowing DB schema
✓  DB schema confirmed before writing API
✗  Voice alert system skipped or stubbed
✓  Dedicated prompt = full voice implementation
✗  50% complete features across the board
✓  Each phase = fully working, testable module
✗  Hard to debug — everything is tangled
✓  Modular = isolated bugs, easy fixes
✗  Agent asks 10 clarifying questions mid-build
✓  All context front-loaded in Phase 0
✗  You get spaghetti code
✓  You get clean, layered architecture
Rule of thumb: If your app has more than 3 features or involves a database + backend + frontend, always break. SafeGuard AI has 14+ major features — breaking into 6 phases is non-negotiable for quality output from any AI coding agent.

PHASE 1 — AI Detection Engine (YOLOv8 + PPE Detection Pipeline)
Only send after Phase 0 passes health check. This builds the core AI brain — the video processing and PPE detection pipeline.
PROMPT 1   AI Detection Engine
Phase 0 is complete. Now build the AI detection engine for SafeGuard AI.
 
DETECTION SERVICE: ai_engine/detector.py
  Use ultralytics YOLOv8 (pip install ultralytics).
  Load model: yolov8m.pt as base, we will fine-tune later.
  For now, use COCO classes to detect: person(0), backpack-for vest approx.
  Create a custom class mapping file: ai_engine/ppe_classes.yaml
    - 0: helmet
    - 1: no_helmet
    - 2: safety_vest
    - 3: no_vest
    - 4: safety_gloves
    - 5: no_gloves
    - 6: safety_goggles
    - 7: safety_boots
    - 8: body_harness
 
  Detection pipeline:
    1. Accept RTSP stream URL or local video file path
    2. Read frames at 30 FPS using OpenCV
    3. Run YOLOv8 inference on each frame (batch=1)
    4. For each detected person, check which PPE is present/missing
    5. If ANY required PPE for the zone is missing with confidence > 0.87:
       - Extract worker face ROI from frame
       - Emit violation event to Redis channel 'violations'
       - Save annotated frame to MinIO bucket 'violations'
    6. Return annotated frame with bounding boxes and labels
 
CAMERA MANAGER: ai_engine/camera_manager.py
  - Manage multiple RTSP streams concurrently using asyncio
  - Each camera runs in its own async task
  - Support up to 16 concurrent streams per node
  - Reconnect automatically on stream drop (max 3 retries, then alert)
  - Associate each camera_id with a zone from the zones table
 
API ENDPOINTS to add in backend/api/detection.py:
  POST /cameras/start  - start monitoring a camera RTSP stream
  POST /cameras/stop   - stop a camera stream
  GET  /cameras        - list all active cameras and their status
  GET  /violations     - paginated list of violations with filters
  GET  /violations/{id} - single violation with image URL
  WebSocket /ws/live/{camera_id} - stream annotated frames as JPEG
 
PERFORMANCE REQUIREMENTS:
  - Detection latency: < 200ms per frame (use GPU if available, CPU fallback)
  - Use ONNX export of YOLOv8 for CPU deployments
  - Frame queue max size: 5 (drop oldest if full to maintain real-time)

PROMPT 2   Worker Identity & Facial Recognition
Phase 1 is complete and detecting violations. Now build the worker identity system.
 
FACIAL RECOGNITION ENGINE: ai_engine/face_recognition.py
  Use insightface library (pip install insightface onnxruntime)
  Model: buffalo_l (ArcFace ResNet100 — best accuracy)
  Initialize: app = FaceAnalysis(name='buffalo_l')
 
  Functions to implement:
    enroll_face(worker_id, image_bytes) -> embedding (512-dim numpy array)
      - Detect face in image
      - Extract 512-dimensional ArcFace embedding
      - Normalize embedding (L2 norm)
      - Store as JSONB in workers.face_embedding
      - Return success/failure with confidence
 
    identify_face(face_image_bytes, threshold=0.45) -> {worker_id, name, confidence}
      - Extract embedding from query face
      - Load all embeddings from workers table (cache in Redis with 5min TTL)
      - Compute cosine similarity against all enrolled workers
      - Return best match if similarity > threshold, else 'unknown'
      - Performance target: < 50ms for 10,000 workers
 
  Integration with detection pipeline:
    When a violation is emitted to Redis 'violations' channel:
      1. Extract face ROI from the violation frame
      2. Run identify_face()
      3. Enrich the violation record with worker_id (or null if unknown)
      4. Update worker compliance_rate in workers table
 
WORKER PROFILE API: backend/api/workers.py
  POST /workers                     - create worker (name, employee_code, language)
  POST /workers/{id}/enroll-face    - upload face photo for enrollment
    - Accept multipart/form-data image
    - Run enroll_face(), store embedding
  GET  /workers                     - list workers with compliance stats
  GET  /workers/{id}               - worker detail + violation history
  GET  /workers/{id}/safety-score  - current week compliance percentage
  PUT  /workers/{id}               - update worker details
  DELETE /workers/{id}/face        - remove face embedding (right to erasure)
 
PRIVACY RULES (implement strictly):
  - Never store actual face images — only 512-dim embeddings
  - Add is_privacy_zone boolean to zones table
  - In privacy zones: skip face identification entirely
  - All embeddings encrypted at rest using Fernet (cryptography library)

PROMPT 3   Alert Engine & Voice System (23 Languages)
Phase 2 is complete. Now build the real-time alert engine and voice system.
This is the most critical differentiator of SafeGuard AI — voice alerts for
illiterate workers in their regional language.
 
ALERT ENGINE: alert_engine/alert_processor.py
  Subscribe to Redis channel 'violations'
  Implement 6-level escalation ladder:
 
  LEVEL 1 — Instant Visual + Voice (T+0ms)
    - Trigger: violation detected with confidence > 0.87
    - Action: emit WebSocket event to dashboard for flash overlay
    - Action: call VoiceAlertService.announce(zone, worker, missing_ppe)
    - Store alert in alerts table (level=1)
 
  LEVEL 2 — Supervisor Push Notification (T+5 seconds)
    - If violation still unresolved after 5 sec
    - Action: Send FCM push to supervisor mobile app with:
      {worker_name, zone, ppe_missing, camera_screenshot_url, timestamp}
    - Store alert in alerts table (level=2)
 
  LEVEL 3 — Wristband Vibration (T+30 seconds)
    - If worker is enrolled and has registered wristband
    - Action: Publish MQTT message to topic workers/{worker_id}/wristband
      Payload: {vibrate: 3, beep: true, duration_ms: 500}
    - Store alert in alerts table (level=3)
 
  LEVEL 4 — Supervisor Auto-Call (T+90 seconds)
    - Action: Use Twilio API to auto-call supervisor's registered number
    - Play TwiML: 'PPE violation in Zone {zone}. Worker {name}. Immediate attention required.'
    - Store alert in alerts table (level=4)
 
  LEVEL 5 — Site Manager Dashboard Flag (T+3 minutes)
    - Action: Set violation.is_escalated=true
    - Emit WebSocket event to site-manager channel
    - Store alert in alerts table (level=5)
 
  LEVEL 6 — Access Lockout (configurable, T+3 minutes)
    - If zone has access_control=true AND violation persists
    - Action: POST to zone.lockout_endpoint to trigger turnstile/barrier
    - Store alert in alerts table (level=6)
 
VOICE ALERT SERVICE: alert_engine/voice_service.py
  Use gTTS (pip install gTTS) for TTS in multiple languages.
  Support these languages initially (expand to 23 via Azure TTS in production):
    hi: Hindi, te: Telugu, ta: Tamil, bn: Bengali, mr: Marathi,
    kn: Kannada, gu: Gujarati, pa: Punjabi, or: Odia, ml: Malayalam
 
  ALERT MESSAGE TEMPLATES (translate all to each language):
    helmet_missing: 'Attention {name}. Please wear your helmet immediately.
                     Helmet required in this zone.'
    vest_missing:   'Attention {name}. Please wear your safety vest.
                     Vest required before entering this zone.'
    gloves_missing: 'Attention {name}. Please wear your safety gloves.
                     Gloves are mandatory here.'
    generic:        'PPE violation detected in Zone {zone}.
                     A supervisor has been notified.'
    morning_brief:  'Good morning {name}. You have {stars} safety stars
                     this week. Remember your {missing_ppe} today.'
 
  announce(zone_id, worker, missing_ppe_list):
    1. Get worker language preference (default: 'hi')
    2. Generate appropriate message template
    3. Use gTTS to generate MP3 audio
    4. Cache audio in Redis for 1 hour (same alert repeated = use cache)
    5. POST audio file to PA system HTTP endpoint OR
       play via pygame if local speakers connected
 
  MORNING BRIEFING FLOW (run at 06:30 site timezone daily):
    For each expected worker (based on schedule or all enrolled workers):
      1. Generate personalized morning message in their language
      2. Queue for kiosk playback when they check in
 
MQTT BROKER: alert_engine/mqtt_manager.py
  Use paho-mqtt to connect to Mosquitto broker (add to docker-compose)

    workers/{worker_id}/wristband  -> vibration commands

4. How to Use These Prompts — Rules for the Agent
GOLDEN RULES — Follow These or Your Build Will Break
R1
Never send Phase N+1 until Phase N is fully working and tested. Test manually before pasting the next prompt.
R2
At the start of EVERY phase prompt, paste the last 10 lines of your current folder structure so the agent has context on what already exists.
R3
If the agent goes off-script (e.g., uses Flask instead of FastAPI, or SQLite instead of PostgreSQL), stop it immediately and correct before continuing.
R4
Always ask the agent: 'What files did you create or modify in this phase?' and verify against the expected deliverable before proceeding.
R5
Keep a running PHASE_LOG.md file listing: what was built, what works, what is pending. Paste a summary into the next phase prompt as context.
R6
If a phase produces an error, do NOT start the next phase. Fix the error first. Errors compound — a broken Phase 1 breaks all subsequent phases.
R7
For the kiosk UI (Phase 4), test on an actual touchscreen or tablet. What looks fine on desktop often has touch target sizing issues on device.
R8
After all 6 phases are complete, do one final prompt: 'Review the entire codebase for security issues: SQL injection, missing auth on endpoints, unencrypted embeddings, exposed env vars. Fix all issues found.'
